### Day 1 里程碑验收

完成 Sprint 1~6 后，Day 1 版本达到以下标准：

- [ ] **核心闭环**: 用户能通过桌面应用创建、浏览、编辑、删除、搜索知识
- [ ] **AI 接入**: Claude Code 等工具能通过 `memoforge serve` 的 MCP 接口读写知识
- [ ] **Git 集成**: 用户能 init / commit / pull / push，冲突时返回详细错误
- [ ] **跨进程同步**: MCP Server 创建的知识在桌面应用中自动出现
- [ ] **冷启动**: 新用户能顺利完成首次使用流程（含基础模板选择）
- [ ] **错误处理**: 所有接口返回统一的 `MemoError` 结构，AI Agent 能识别和处理

## 6. 关键技术风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| Milkdown 编辑器集成复杂度 | 可能影响 Sprint 5 进度 | 可先用简单的 textarea + 预览模式替代，后续替换 |
| git2-rs 在 Windows 上的编译问题 | 跨平台兼容 | 优先支持 macOS/Linux，Windows 后续测试修复 |
| MCP 协议规范变化 | tool 注册方式可能调整 | 关注 MCP spec 更新，保持 tool 层与核心逻辑解耦 |
| 多进程文件锁在 NFS/网络文件系统不可靠 | 网络挂载的知识库可能锁失败 | 文档说明仅支持本地文件系统，NFS 不保证并发安全 |
| Frontmatter 索引缓存内存占用 | 大型知识库（>10K条）可能内存高 | 索引只缓存 L0 级别数据，每条约 200 bytes |

## 7. 测试策略

### 7.1 单元测试

每个 Sprint 的核心模块都需要单元测试覆盖：

| 模块 | 测试重点 |
|------|---------|
| frontmatter.rs | 解析/渲染正确性、边界情况（空 frontmatter、缺失字段） |
| knowledge.rs | CRUD 流程、路径校验、dry_run 行为 |
| search.rs | 正则匹配、上下文行数、大小写 |
| git.rs | init/commit/push/pull、冲突处理、错误码映射 |
| lock.rs | 非阻塞冲突返回、批量锁字典序、全局写锁共享/排他 |
| error.rs | 错误码覆盖、序列化格式 |

### 7.2 集成测试

- **MCP 端到端测试**: 启动 `memoforge serve`，通过 stdio 发送 JSON-RPC 请求，验证响应
- **Tauri IPC 测试**: 前端调用 Tauri command，验证数据正确传递
- **跨进程测试**: MCP Server 写入 → GUI 文件监听 → UI 自动刷新

### 7.3 手工验收测试

每个 Sprint 完成后进行手工验收：
- 按 UI 设计稿（Pencil 5 页）逐页比对
- 在 Claude Code 中实际配置 MCP Server 并使用
- 模拟新用户冷启动流程

### 7.4 CI/CD 集成

建议在 Sprint 1 完成后即配置 GitHub Actions：

- **基础流水线**：`cargo build --workspace` + `cargo test --workspace` + `cargo clippy`
- **前端检查**：`npm run lint` + `npm run typecheck`
- **触发条件**：PR 提交和 main 分支推送
- **后续扩展**：Tauri 构建（需 macOS/Linux/Windows 三平台 runner）、MCP 端到端测试
