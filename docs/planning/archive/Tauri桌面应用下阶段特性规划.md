## 1. 文档目的

本文件聚焦 MemoForge 桌面应用本身的下一阶段增强，不讨论 MCP 协议细节本身，而是讨论：

- 从用户视角，桌面应用当前卡在哪里
- 从产品视角，下一阶段应该优先做什么
- 从实现视角，Tauri + React 前端需要怎样落地
- 从测试视角，哪些桌面链路必须补齐

这份文档与 Agent/MCP 方案互补：

- `Agent协作增强与MCP精简方案` 解决 Agent 如何更稳定地写入知识
- 本文解决桌面应用如何更好地承载知识工作流、初始化流程、日常维护和 Agent 结果消费

---

## 2. 当前状态判断

MemoForge 桌面端已经具备以下核心能力：

- 多知识库切换
- 导入 Markdown 文件夹
- 阅读 / Markdown / 富文本三种编辑模式
- 分类树和目录浏览
- 搜索、图谱、Git 面板
- 设置页、日志诊断、MCP 连接状态

这意味着当前桌面端已经不再是“壳”，而是一个可用产品。

但从真实用户体验看，仍存在三类明显缺口：

1. 首次使用成功率不够高
2. 长期使用的整理体验不够强
3. Agent 写入结果的消费与确认体验不够完整

---

## 3. 用户视角的主要问题

### 3.1 冷启动路径不完整

当前界面已有知识库切换与“打开 / 初始化”能力，但冷启动更像工程入口，而不是产品入口：

- 用户仍要自己理解“打开知识库”和“初始化知识库”的区别
- 缺少显式的首次使用向导
- 缺少从模板开始的可见入口
- 缺少 Git clone 的桌面流

### 3.2 导入后整理没有闭环

已有导入能力，但导入结束后用户还需要自己处理：

- 哪些文档没有摘要
- 哪些文档没有 tags
- 哪些目录结构不理想
- 哪些知识是孤立的

导入解决了“带进来”，没有解决“带进来之后怎么变成高质量知识库”。

### 3.3 日常使用缺少强任务视图

当前界面主要围绕“浏览”和“编辑”组织，但长期使用时用户更需要：

- 最近编辑
- 最近被 Agent 更新
- 待整理文档
- 摘要过期
- 无标签 / 无分类 / 孤立知识

也就是从“文件浏览器”进一步升级为“知识工作台”。

### 3.4 Git 能力有了，但协作体验还不够强

Git Panel 现在已经可以提交、拉取、推送，但在用户感知上仍偏底层：

- 远程仓库状态不可见
- pull 前没有差异预警
- push 失败后的恢复建议不够强
- 团队协作中“谁改了什么”不够可见

### 3.5 设置页还偏诊断，不是操作中心

当前设置页已经有日志和 MCP 配置帮助，这很好，但还不够：

- 没有“知识库偏好”入口
- 没有“默认编辑模式 / 自动保存 / 导入策略”等用户级配置
- 没有“我的工作流”相关设置

## 4. 产品目标

### 4.1 下阶段桌面端的产品目标

1. 把桌面端打造成真正的知识工作台，而不是 Markdown 容器。
2. 把首次使用从“技术操作”改成“产品引导”。
3. 把 Agent 结果消费、知识整理和 Git 协作纳入统一桌面流程。
4. 在不破坏现有轻量体验的前提下，让核心高频操作更顺手。

### 4.2 非目标

本阶段不做：

- Web 端完整编辑器能力
- 多人实时在线协作
- 复杂权限体系
- 内嵌聊天模型
- 复杂插件系统

## 5. 下一阶段特性建议

### P0: 首次使用与知识库入口重构

这是最优先的桌面端工作。

#### 5.1 首次使用向导

新增启动页或欢迎流，明确三条路径：

1. 创建新知识库
2. 导入已有 Markdown 目录
3. Clone Git 仓库

每条路径都要有：

- 一句话说明
- 适用对象
- 结果预期

#### 5.2 模板知识库入口

在“创建新知识库”流程中提供模板：

- 开发者知识库
- 项目复盘模板
- 技术阅读笔记模板

模板价值不是内容多少，而是帮助新用户一打开就理解“应该怎么组织”。

#### 5.3 知识库管理中心升级

把当前 `KbSwitcher` 升级为“知识库管理中心”：

- 最近打开
- 本地路径健康状态
- 远程仓库状态
- 重新打开 / 移除注册

说明：

- “默认知识库”“重新索引 / 修复”需要先补齐 registry 与维护能力，暂不作为本阶段管理中心的首批验收项

### P0: 导入体验升级

#### 5.4 导入后整理报告

导入完成后不只显示数量，而是显示：

- 新增文档数
- 生成 frontmatter 数
- 未生成摘要数
- 未分类文档数
- 推荐处理动作

#### 5.5 导入任务收尾动作

导入结束后提供按钮：

- 去补摘要
- 去补标签
- 去整理分类
- 去查看新导入文档

### P1: 知识工作台

#### 5.6 首页 / 工作台视图

建议新增一个“工作台”视图，展示：

- 最近编辑
- 最近打开
- 最近导入 / 最近移动
- 摘要待更新
- 待补标签
- 孤立知识

这个页面应成为长期用户打开 MemoForge 后的默认落点之一。

说明：

- “最近被 Agent 更新”适合在 Agent Draft 流和事件聚合稳定后再补入

#### 5.7 知识健康度

增加一套知识健康度指标：

- 无摘要
- `summary_stale`
- 无标签
- 无分类
- 无入链 / 无出链
- 标题与文件名明显不一致

桌面端应该把这些问题变成可处理的清单，而不是靠用户自己发现。

### P1: 搜索与检索增强

#### 5.8 搜索结果组织升级

当前搜索已经能用，但可以更产品化：

- 按文档聚合结果
- 显示匹配 section
- 支持“最近度 / 相关度 / 目录”排序
- 支持“只看摘要 / 只看正文 / 只看标题”

#### 5.9 快速动作

在搜索结果和文档列表上增加快捷动作：

- 复制路径
- 复制 wiki link
- 加入相关知识
- 标记待整理

### P1: Agent 结果消费

#### 5.10 Agent 草稿面板

与 Agent/MCP 文档联动，桌面端应提供：

- 待确认草稿列表
- 草稿预览
- 提交 / 丢弃
- 最近 Agent 活动

#### 5.11 Agent 写入回显

在文档内或侧栏显示：

- 最后更新来源: GUI / MCP / Claude Code / OpenCode
- 最后更新时间
- 最近一次变更摘要

### P2: Git 协作体验

#### 5.12 Git 状态升级

Git 面板补充：

- 当前分支
- 远程追踪状态
- ahead / behind
- 未推送提交数

#### 5.13 Push/Pull 保护

在关键操作前增加提示：

- pull 前提示本地未提交变更
- push 前提示远端领先风险
- 冲突时显示恢复建议

#### 5.14 变更来源时间线

在桌面端提供最近事件流：

- GUI 编辑
- Agent 写入
- Git commit / pull / push
- 导入 / 移动 / 删除

### P2: 设置与偏好中心

#### 5.15 设置页升级

建议把设置分组：

- 通用
- 编辑器
- 知识库
- Git
- MCP / Agent
- 诊断与日志

#### 5.16 重点偏好项

建议新增：

- 默认编辑模式
- 自动保存间隔
- 新建知识默认分类
- 导入默认策略
- 搜索默认模式
- 是否自动打开最近知识库

## 6. 桌面端实现建议

### 6.1 Tauri 后端新增 command 建议

建议补充或重构以下 command：

- `clone_kb_cmd`
- `list_templates_cmd`
- `create_kb_from_template_cmd`
- `get_workspace_overview_cmd`
- `get_kb_health_cmd`
- `get_recent_activity_cmd`
- `get_git_overview_cmd`
- `repair_kb_index_cmd`

### 6.2 前端新增组件建议

建议新增：

- `WelcomeFlow.tsx`
- `TemplatePicker.tsx`
- `WorkspaceDashboard.tsx`
- `KnowledgeHealthPanel.tsx`
- `RecentActivityPanel.tsx`
- `GitOverviewPanel.tsx`

建议改造：

- `KbSwitcher.tsx`
- `ImportModal.tsx`
- `SettingsModal.tsx`
- `SearchPanel.tsx`
- `RightPanel.tsx`

### 6.3 状态管理建议

在 Zustand 中新增两类状态：

- `workspaceOverview`
- `knowledgeHealth`

避免把这些聚合逻辑全部堆在 `App.tsx`。

## 7. 测试方案

### 7.1 桌面端主路径 E2E

必须覆盖：

1. 首次打开欢迎流
2. 新建知识库
3. 从模板创建知识库
4. 导入 Markdown 目录
5. Clone Git 仓库
6. 多知识库切换
7. 工作台数据刷新
8. Git push/pull 风险提示
9. Agent 草稿可见与确认

### 7.2 组件测试

至少覆盖：

- `KbSwitcher`
- `ImportModal`
- `SettingsModal`
- `WorkspaceDashboard`
- `KnowledgeHealthPanel`
- `GitOverviewPanel`

### 7.3 回归重点

必须确保不回退：

- 已有编辑模式
- 搜索功能
- 图谱打开与选中
- 导入流程
- Git 提交与同步
- 只读 Web 模式

## 9. 团队建议

如果采用 Agent Teams 并行开发，建议在桌面端侧单独配一条产品线：

- `Desktop Product Lead`
  - 负责桌面端用户路径、优先级、验收标准
- `Tauri Backend Worker`
  - 负责 command、状态聚合、诊断能力
- `Frontend UX Worker`
  - 负责欢迎流、工作台、设置页、健康度面板
- `Git / Workspace Worker`
  - 负责 Git 总览、知识库管理中心、模板 / clone 流
- `QA Worker`
  - 负责桌面端主路径 E2E 和组件回归

---

## 10. 验收标准

满足以下条件，视为桌面端下阶段增强完成：

1. 新用户能在 3 分钟内完成“新建 / 导入 / clone”任一路径。
2. 用户能在桌面端快速发现待整理知识，而不是靠手工巡检。
3. Git 状态和风险对用户是可理解、可操作的。
4. Agent 的写入结果在桌面端是可见、可确认、可追踪的。
5. 设置页从诊断入口升级为真正的偏好中心。
